# MyJara Platform Stabilization & Feature Roadmap

## Context

MyJara has gone through several rapid AI-assisted development passes. That left behind duplicate/conflicting database schemas (three different chat table sets, two migration directories that disagree with each other and with `src/types/database.ts`), admin tooling that queries columns/tables that don't exist, a paywall that's only half-enforced, and — most urgently — a `users` RLS policy that exposes every user's DOB, home address, bank account number, and tax ID to anyone, unauthenticated. On top of that unstable foundation, there's a long list of real product gaps: wholesalers can't see or monetize their own stores properly, categories are split across two disconnected taxonomies, chat doesn't support product references or admin-user conversations reliably, and the UI has no visual identity beyond flat Tailwind defaults.

Given the size of this (five distinct workstreams, dozens of individual items), the goal here is not to fully spec every feature today. It's to sequence the work by risk — stop active data corruption/exposure first, then make already-half-built features (chat, admin visibility, wholesaler dashboard) actually work, then build the new monetization/ads/UI initiatives on top of a stable base — and to fully scope the phase we start with (Phase 0) so it's immediately executable. Phases 1–4 are roadmap-level scoping; each gets its own detailed implementation pass when we reach it.

This plan reflects two rounds of live code investigation (not just the docs), so some things flagged during scoping turned out to be worse than described (chat has three incompatible schemas, not just "not working"; admin/messages references a table that doesn't exist at all) and a couple turned out to be partially built already (the category multi-select on seller profile-edit *does* save — it just saves to a taxonomy nothing else reads, which is likely why it feels broken).

**Status:** Phase 0 (0.1–0.5), Phase 1 (1.1–1.5), and Phase 2.1 + 2.2 + 2.4 + 2.6 have been implemented in code. 2.3 is done except the wholesaler per-conversion billing engine (blocked on billing-mechanics answers). 2.5's core (multi-location management + storefront display) is built; checkout/product location wiring is deferred to a follow-up pass — see `supabase/migrations/021`–`034` and `supabase/migrations/README.md`. Migrations 021–024 (Phase 0) have been applied to the live database via the Supabase SQL Editor. Migrations 025–034 (Phase 1, 2.1, 2.2, 2.3, 2.5, 2.6, Phase 4 legal pages) are written but **not yet applied** — same process, run them in order via the SQL Editor. Phase 2.6 also needs a `CRON_SECRET` env var set in the Vercel project (Vercel Cron sends it automatically as a Bearer header once configured) for `/api/cron/subscription-reminders` to run. Confirmed with the site owner: canonical domain is `myjara.com.ng` (swept across the codebase), retailer plans should differ by shop_type (now supported via `/admin/pricing`). **Still blocking: the wholesaler per-conversion billing mechanics (conversion definition, rate, collection method) — see open questions.** All Phase 2 items are now started. Phase 4 is well underway: `Sheet`/`Toast` match `dialog.tsx`'s frosted-glass treatment, the homepage category grid is removed, `/how-jara-works`/`/how-it-works` are consolidated, the footer is role-aware, `/terms`/`/privacy` are now admin-editable via `/admin/legal`, admin analytics gained conversion/repeat-purchase-rate/location widgets, and the admin nav bar + dashboard got a visual pass (grouped nav, real Platform Health metrics instead of hardcoded fake ones, fixed a couple of small pre-existing bugs found along the way). Phase 3 (Ads) is unstarted — it's a new monetization feature with its own pricing/targeting decisions, likely blocked on site-owner input the same way wholesaler billing was.

**Also fixed this pass, found while investigating a user bug report of "duplicate retailer dashboards and chat boxes":**
- **Security:** `createCategoryAction`/`updateCategoryAction`/`deleteCategoryAction` (`categories.ts`), `approveWholesalerAction`/`rejectWholesalerAction` (`verification.ts`), and `resolveDisputeAction` (`disputes.ts`) all used the service-role client with **no check that the caller is an admin** — any unauthenticated caller invoking these server actions directly could edit categories, approve/reject any wholesaler, or resolve disputes. All now guarded with `getAdminSession()`. Also deleted `src/app/actions/admin.ts`, an abandoned unguarded duplicate of the verification actions (its own comments documented the confusion); `verification-list.tsx` now calls the real, guarded actions.
- **The actual duplication bug:** the header's profile dropdown and inbox icon (`src/components/shared/header.tsx`) were never role-gated, so retailers/wholesalers could reach `/customer/dashboard` and `/inbox` (built for the customer flow) *in addition to* their real dashboard and messages page — two disconnected views presenting as "the dashboard"/"the inbox" for the same login. Fixed by role-gating both; `/customer/dashboard` now redirects non-customers to their real dashboard.
- Hardened `getOrCreateChatRoomAction`/`getOrCreateChatRoomWithCustomerAction` against a race that surfaced a generic error instead of returning the room actually created.
- **Fixed:** `/seller/support` submitted to a `console.log` + fake delay instead of a real backend. Now calls the existing `createSupportTicketAction` (`disputes.ts`), which writes into the `disputes` table and surfaces in `/admin/disputes` like any other ticket; the form's `type` maps onto the existing `cause` taxonomy, priority is folded into the description text (nothing downstream reads it as a separate field).
- **Not done, needs a human data check first:** the legacy `chat_conversations`/`chat_messages` tables are unreferenced by the app, but migration `023`'s own comment says they should only be dropped after confirming the live rows hold nothing worth preserving — that's a live-data judgment call I can't make without DB access, so no DROP migration was written. If you're fine losing whatever's in there, say so and I'll write the migration.
- **Fixed, found while auditing revenue numbers across every role's dashboard:** `orders.total_amount` and `orders.buyer_id`/`orders.customer_id` don't exist on the real table (`001_initial_schema.sql`: it's `total` and `user_id`) — `admin/transactions/page.tsx`, the brand/customer/seller dashboards, all queried these nonexistent columns, so revenue and recent-orders widgets errored silently and rendered zero/empty everywhere. Also rebuilt `admin/transactions/page.tsx` to query the real `transactions` table (one row per Flutterwave payment) instead of `orders`, added a Store column via the `orders`/`stores` joins, and fixed its status colors to match the real `transaction_status` enum (`pending|success|failed`, not `completed`/`processing`/`cancelled`). Also fixed a duplicate/dead `transactions.insert()` call in `payment-button.tsx` (an earlier broken insert with no `.insert()` chained, followed by the real one — the error check was reading the wrong call's result). Verified via full-repo grep (no other file references the bad column names) and `tsc --noEmit` (clean).

## Phase 0 — Security & Data-Integrity Emergency

No user-visible feature work in this phase — the goal is to stop active harm and give every later phase a foundation that won't silently corrupt data.

**0.1 Fix the `users` RLS PII leak**
`supabase/migrations/001_initial_schema.sql:239-241` has `CREATE POLICY "Public can view basic user info" ON users FOR SELECT USING (true)`. Later migrations added `date_of_birth`, `residential_address`, `bank_name`, `account_number`, `account_name`, `rc_number`, `tax_id_number`, `directors_info` directly onto this table without ever narrowing that policy. Anyone with the anon key can currently read all of it for every user.
- New migration: replace the blanket policy with `USING (auth.uid() = id)` for the full row.
- Grep every client-side `supabase.from('users').select(...)` call (chat participant display, store-owner name lookups, etc.) and either move them server-side or point them at a new narrow `public_user_profiles` view exposing only `id, full_name, avatar_url`.

**0.2 Fix `disputes` RLS + the admin disputes page**
- `disputes` RLS currently only allows `customer_id = auth.uid()` — there's no admin or store-owner read policy at all (the migration's own comment calls it a placeholder). Add one.
- `admin/disputes/page.tsx` uses the anon client instead of `createAdminClient()` like every other admin page — switch it.
- The page selects a `subject` column that doesn't exist on the real table (`create_disputes_table.sql` has `reason`, not `subject`) — fix the query to match actual columns (`reason, description, cause, status`).

**0.3 Consolidate the chat schema (fixes admin↔user chat, chat broken for some users)**
Three incompatible schemas exist today: `chat_rooms`/`messages` (`012_chat_system.sql`, what `src/app/actions/chat.ts` and all real chat UI use), the orphaned `chat_conversations`/`chat_messages` (`001`/`002_chat_and_customization.sql`, still queried by one marketplace chat page but otherwise dead), and a `conversations` table referenced by `admin/messages/page.tsx` that **doesn't exist in any migration** — that page is almost certainly erroring or silently empty right now.
- Standardize on `chat_rooms`/`messages` since it's what's actually wired through the action layer everywhere else.
- Rebuild `admin/messages/page.tsx` on top of the existing `chat.ts` actions (`getChatRoomsAction`, `sendMessageAction`, etc.) instead of the nonexistent `conversations` table.
- Retire the orphaned `chat_conversations` schema once its one remaining caller (`(marketplace)/chat/[id]/page.tsx`) is migrated or removed.

**0.4 Fix admin analytics revenue computation**
`admin/analytics/page.tsx` and the `get_sales_by_location` RPC both filter `orders.status = 'completed'`, but the real `OrderStatus` enum is `pending|paid|processing|shipped|delivered|cancelled` — `'completed'` never matches, so revenue is likely always reporting 0. Fixed to use `delivered` (the status sellers actually set via the "Mark Delivered" action) as the revenue-recognized terminal status.

**0.5 Reconcile the two migration directories**
`supabase/migrations/*.sql` (numbered, looks official) and `src/lib/supabase/migrations/*.sql` (20 ad-hoc scripts that actually add columns/tables the app depends on, e.g. `stores.categories`, `disputes`, the analytics RPC) disagree with each other and with `src/types/database.ts`, which is why the codebase is full of `as any` casts. A full historical merge turned out to be unsafe to do blind (see `supabase/migrations/README.md` for why); instead, new bugs found were fixed as idempotent migrations layered on top of whatever the live state actually is, and a `supabase db pull` / `supabase gen types` follow-up is documented for whoever has live DB credentials.

**Phase 0 verification:** confirm an anon/logged-out request can no longer read other users' PII; log in as admin and successfully view + respond in a real chat conversation via `admin/messages`; confirm `admin/disputes` shows disputes filed by a test account; place a test order through to its terminal status and confirm it shows up in admin analytics revenue.

## Phase 1 — Core Data Model & Admin Visibility

**1.1 Unify the categories taxonomy** — this is the root cause behind the "categories not editable" and "admin can't see products by category" complaints. Sellers pick from a hardcoded `PRODUCT_CATEGORIES` list in `src/lib/constants.ts`, stored as raw strings in `stores.categories`. Products, meanwhile, use `products.category_id`, a real foreign key into the `categories` table (UUID primary keys). These two taxonomies are disconnected — the seller-side multi-select on `seller/profile/edit/page.tsx` genuinely does save (`src/app/actions/profile.ts:83-89`), but it saves into a field nothing else reads, which is likely why it feels broken from the outside.
- Migrate `PRODUCT_CATEGORIES` into real `categories` rows (top-level slugs already match the seeded data in `001_initial_schema.sql`, so this is mostly a mapping exercise), change `stores.categories` to reference `categories.id` (or a `store_categories` join table), and update both the profile-edit page and admin categories page to read/write the same source of truth.
- Build actual CRUD on `admin/categories/page.tsx` (currently view-only — no create/edit/delete for name, slug, or icon).
- Fix a real bug found along the way: `updateProfile()` skips writing `categories` entirely when the array is empty (`if (formData.categories && formData.categories.length > 0)` in `profile.ts:83`), so a seller can never clear all their categories.

**1.2 Admin store & product visibility**
- `admin/stores/page.tsx` queries an `is_verified` column that doesn't exist in any migration — add the column or fix the query — and add a retailer-vs-wholesaler type filter (by `shop_type`).
- Add a "products by category" filter/view to the admin panel.

**1.3 Registration → profile completeness**
- Decide, field by field, which registration-collected values (full name, email, sex, DOB) should become editable vs. intentionally locked (e.g. legal-name changes routed through support), then wire the rest into `seller/profile/edit` and the customer settings page.
- Extend wholesaler/brand registration (`register/brand/page.tsx`) to collect what retailer registration already does — it currently hardcodes `categories: []` and skips several fields the retailer flow captures.

**1.4 User tags** — clarified: every user/vendor/admin should get a short, unique, human-readable tag (distinct from their `id` UUID) so people and disputes can be found without quoting a raw UUID.
- `users.tag`: unique, generated at registration from `slug(name) + '-' + first 4 chars of the real UUID` — keeps the tag deterministically tied back to the UUID (the thing every FK and RLS policy actually uses) without encoding meaning *into* the UUID itself, which would fight its purpose as an opaque key. Admins get an `admin-` prefixed tag.
- Searchable in chat (`searchUsersAction`, `searchStoresAction`) and in the admin disputes list; admin/users shows both `tag` and the full `id` for reference.
- **Deferred, not built yet:** "wholesalers can name up to 5 tagged staff accounts (support/disputes/etc.)" and "multi-shop retailers get one tag per shop, varying by location/category" both need a real sub-account/staff model and the multi-shop data model neither of which exist yet — `stores` is strictly one-row-per-owner today (see 2.1's `store_locations` note and the assumption audit in 2.5). Building tags for accounts that can't exist yet would be premature; revisit as part of 2.1 (wholesaler staff accounts) and 2.5 (multi-shop) once those land, then extend tag generation to encode shop location/category at that point.

**1.5 Chat: product references + dispute flagging** (builds on the 0.3 consolidation)
- Add a nullable `product_id` (or a `metadata jsonb`) to `messages` and a compact product-attach UI in the chat composer, reusing `ProductCard`.
- Add a "flag as dispute" action on a conversation that creates a `disputes` row linked to it, visible only to the admin and that conversation's user.

## Phase 2 — Wholesaler / Multi-Tenant Infra & Monetization

**2.1 Wholesaler dashboard fixes — done**
- Fixed the hardcoded "View Store" link, added settlement-account fields and a `/dashboard/wallet` page mirroring the retailer one.
- Closed the `/store/[slug]/admin` per-store access-key backdoor entirely (route deleted, `admin_access_key` column dropped in migration 029) and secured the global `/admin` panel with real per-admin Supabase accounts alongside the existing master `ADMIN_SECRET_KEY` (kept as an intentional fallback).
- Also fixed while in this area: three broken chat-initiation entry points — the storefront had no "Chat with Vendor" button at all, store-side inboxes (seller/brand support & messages, admin messages) had no way to search someone new and start a conversation, and the wholesaler order-detail "Chat with Customer" button linked to a dead query param. All now use the same `getOrCreateChatRoomWithCustomerAction` / `searchUsersAction` primitives that already existed but were unwired.

**2.2 Custom domain connection — done** — built on the existing, previously-unused `store_domains` table, which had RLS enabled with zero policies (nothing, not even the owning store, could read or write it) — migration 031 adds owner CRUD policies, a narrow public-read policy for verified domains only, and a `verification_token` column. New "Custom Domain" card on the wholesaler dashboard settings page (`dashboard/settings/custom-domain-card.tsx`) lets an owner add a domain, shows DNS instructions (CNAME/A to Vercel + a `_myjara-verify` TXT record carrying a random per-domain token), and a Verify button that does a live DNS TXT lookup server-side (`src/app/actions/domains.ts`) before flipping `is_verified`. `src/lib/supabase/middleware.ts` now resolves verified custom-domain hostnames to a store via `store_domains` and rewrites to `/store/[slug]`, the same as subdomain routing.
- **Real pre-existing bug fixed in the same file:** subdomain routing (`brandname.<domain>`) checked `hostname.includes('myjara.com')`, but every other functional link in the codebase (order/subscription emails, action links) used `myjara.ng`. **Resolved:** the site owner confirmed the actual domain is `myjara.com.ng` — swept across the whole codebase (middleware, emails, Flutterwave logo URLs, settings copy), matching what `robots.ts`/`sitemap.ts` already had right.

**2.3 Differentiated pricing & a real paywall — done except the wholesaler billing engine (blocked, see open questions)**
- **Bigger-than-scoped finding:** investigating this surfaced a second, entirely dead subscription subsystem — a `subscriptions` table + `createSubscriptionAction`/`checkSubscriptionStatus` (`app/actions/subscription.ts`) + the old `/dashboard/subscription/page.tsx` — that read/wrote columns (`current_period_end` vs. the page's `end_date`) that didn't even match each other, and had zero real callers (`createSubscriptionAction` was imported in `register/retailer/page.tsx` but never called). Deleted both dead functions and the dead import. The real, live system was always `stores.subscription_plan`/`subscription_expiry` (set at registration, updated by `/seller/subscription`'s Flutterwave/promo flow) — this is what `SubscriptionManager` and the middleware gate now consistently use.
- Extracted the working retailer purchase flow (Flutterwave + promo code) out of `/seller/subscription/page.tsx` into a shared `src/components/subscription/subscription-manager.tsx`, reused by both `/seller/subscription` and the now-fixed `/dashboard/subscription` (previously the broken page above). This is what makes extending the paywall to wholesalers safe — they now have a working renewal path before being gated.
- Extended the middleware subscription-expiry gate to `/dashboard/*` (excluding `/dashboard/subscription` itself), mirroring the existing `/seller/*` gate — wholesalers were previously not paywalled at all regardless of subscription status.
- Reconciled the third, dead plan definition in `register/brand/page.tsx` (a local `SUBSCRIPTION_PLANS` const, never actually referenced by the registration payload, which hardcodes `subscription_plan: 'pro'`) by deleting it — nothing pointed to it.
- **Done: admin-editable, per-shop_type pricing.** Confirmed by the site owner: retailer plans should differ by shop_type, and prices should be admin-editable rather than hardcoded. Moved `SUBSCRIPTION_PLANS`/`WHOLESALER_PLANS` out of `constants.ts` into a `subscription_plans` table (migration 032, seeded with the prior hardcoded prices so this is a pure refactor), scoped per `shop_type` (`physical`/`online`/`market_day`/`brand`). New `/admin/pricing` page (`PricingManager`) lets admins create/edit/hide/delete plans per shop type. `SubscriptionManager` and the retailer registration plan-picker both now read live from the DB instead of the old constants.
- **Still blocked:** the wholesaler billing model. Confirmed by the site owner it should be per-customer-conversion/per-completed-order, not the flat `WHOLESALER_PLANS` tiers (which are still live today as a stopgap so wholesalers aren't unpaywalled). Three mechanics questions are still open before this can be built — see below.
- **Found, flagged, not touched:** Flutterwave client-side code consistently reads `process.env.NEXT_PUBLIC_FLW_PUBLIC_KEY` (`seller/subscription` via `SubscriptionManager`, `components/marketplace/payment-button.tsx`), but `.env.local` only defines `FLUTTERWAVE_PUBLIC_KEY` (no `NEXT_PUBLIC_` prefix, which Next.js requires for any client-bundle-visible env var). If Vercel's prod env doesn't separately have `NEXT_PUBLIC_FLW_PUBLIC_KEY` set, card payments are silently unavailable everywhere (checkout and both subscription pages) and only the promo-code path works — this needs verifying in the Vercel dashboard, not a code fix.

**2.4 Post-signup onboarding flow — done** — built the first real `dialog.tsx` primitive (`@radix-ui/react-dialog` was already a dependency, only wired up for `Sheet` before now) with a frosted-glass treatment, pulled forward from Phase 4 since this phase needed it first. After a successful wholesaler registration (`register/brand/page.tsx`), a `BrandOnboardingDialog` shows a brand-color picker, then opens `/store/[slug]` and `/dashboard` in new tabs.
- **Bug found and fixed along the way:** `registerBrand` creates the auth user server-side via `admin.auth.admin.createUser`, which never establishes a client session — the dashboard tab would have opened straight to `/login` with no way back without re-entering credentials. Fixed by calling `supabase.auth.signInWithPassword()` client-side with the same email/password right after a successful registration, before showing the dialog. (`/store/[slug]` needed no such fix — it already has no `status = 'active'` gate, so a brand-new `pending` store is already publicly viewable, just empty of products.)
- Added `themeColor` to `updateProfile()` (`stores.settings.theme.primaryColor`), merged through the same settings-fetch as `socialLinks` so setting both in one call can't clobber one with a stale read of the other (a latent bug in the original single-key merge, fixed while extending it).

**2.5 Physical retailer multi-shop — core built; checkout/product location wiring deferred**
- **Two competing precedents already existed in the code, and only one survives.** (A) `store_logistics` (`001_initial_schema.sql:101-112`) was already a working, multi-row-per-store child table (`location_name`, `city`, `delivery_fee`, `coverage_zones`) rendered on the storefront and chosen at checkout (`orders.logistics_option_id`) — ~90% of a `store_locations` table already, just scoped to fulfillment rather than full location identity. (B) A separate, *abandoned* scaffold (`src/lib/store-context.ts`, `store-switcher.tsx`, `SellerStoreProvider`) queries `stores` by `owner_id` without `.single()` and lets a user pick an "active store" via cookie — but the "create a second store" flow it depends on (`/onboarding/store`, referenced 5 times) doesn't exist anywhere in `src/app`, so no retailer could ever actually reach a second `stores` row. Built on (A)'s pattern; did not resume (B) — chat (`UNIQUE(user_id, store_id)`) and the public storefront (one page per `slug`) both want one identity per retailer, which (B) would have fragmented.
- **Built:** `store_locations` table (migration 033: `id`, `store_id` FK, `name`, `address`, `city`, `latitude`, `longitude`, `phone`, `gallery_urls`, `is_primary`, `is_active`, `location_type` physical/market_day, `market_name`), RLS (owner CRUD + public read of active locations), and a backfill that gives every existing store one `is_primary` location from its current `stores.market_name/latitude/longitude/phone/gallery_urls`. Those `stores` fields stay in place as the "primary location" mirror (many read sites depend on them directly) rather than migrating every read site in one pass — `src/app/actions/locations.ts` keeps them in sync whenever the primary location is edited or changed, so single-location read paths don't drift.
- **Built:** a "Shop Locations" card on `seller/profile/edit` (physical retailers only, gated by `shop_type`) to add/edit/remove/promote-to-primary locations (`src/components/seller/locations-manager.tsx`), and a "Shop Locations" section on the public storefront (`/store/[slug]`) listing every active location with a map link, shown only when a store has more than one.
- **Re-scoped, not a bug after all:** the original scoping pass flagged `profile.ts`/`verification.ts` writing via `.eq('owner_id', ...)` with no location scoping as a risk "once multi-location ships." Since the shipped design keeps exactly one `stores` row per owner (locations are children of it, not siblings), that risk doesn't materialize — `owner_id` still uniquely identifies one store, so no fix was needed there.
- **Deferred, needs a product decision, not just engineering:** mapped the full checkout flow before touching it (both `checkout-form.tsx` and `cart-checkout-form.tsx` insert orders directly client-side, no server action). `store_logistics.location_id` (added, nullable, in migration 033) still isn't wired in — checkout shows every `store_logistics` row for a store flattened together with no per-location grouping, and `orders` has no location reference at all (only `logistics_option_id` + a free-text address). Actually wiring this means deciding real checkout UX: does a customer pick a specific shop location before seeing its logistics options? Do multi-location stores need to duplicate `store_logistics` rows per location, or does one set of options serve all locations? That's a product call, not something to guess at given it's the live payment path — flagged as an open question below rather than built.
- **Fixed while mapping the above (unrelated to location_id, a real pre-existing bug):** product-level `pickup_location` was read/written from three different, disagreeing places — `seller/products/new` wrote it as a top-level `products` column that doesn't exist (silently dropped or errored), `seller/products/[id]/edit` wrote it correctly under `attributes` but clobbered the product's other attributes with the *store's* settings object on every save, and the customer-facing product page read the same nonexistent top-level column — so a seller's pickup location was never actually visible to a buyer regardless of which form set it. All three now consistently use `attributes.pickup_location`, and edit now preserves the product's own prior attributes instead of overwriting them.

**2.6 Social links & notifications — done** — social links (facebook/instagram/twitter/whatsapp) live in `stores.settings.social`, a JSONB merge alongside the existing `theme` key rather than new columns, editable from both retailer profile-edit and wholesaler dashboard settings, shown as icon links on the storefront. Built a `notifications` table (migration 030) + bell UI in the main header and admin shell, wired into new chat messages and verification decisions. Added a daily Vercel Cron job (`/api/cron/subscription-reminders`) emailing + notifying store owners at 3/1/0 days before `subscription_expiry`.

## Phase 3 — Ads Platform

- New tables: `ads` (store_id, product_ids[], target criteria, status, payment_ref, rejected_reason) linked to `disputes` via a nullable `ad_id`.
- Vendor "Run Ads" button on the brand dashboard: product picker + targeting form + Flutterwave-or-promo-code submission, reusing the existing `checkout-form`/`payment-button` patterns.
- New admin ad review queue: approve → admin controls placement; reject → opens a chat with that vendor (Phase 1.5) scoped to that ad, with one-click dispute flagging.
- Super-admin placement pipeline: an `ad_placements` config (slot key → active ad ids) read by the homepage and other pages at render time, plus a small admin-editable settings table for homepage copy/banners (not a full CMS).
- Per-user signal tracking: derive most-purchased product/category per user from `order_items`, kept server-side only (never exposed in the customer-facing account UI), consumed by the ad-placement query to bias which ads a given user sees on the homepage.
